üìò Exam Line: NFV virtualizes network services such as firewalls, routers, and load balancers using software.

3Ô∏è‚É£ NFV in Practice: How Cloud Providers Offer Network Functions as Services

Modern cloud providers implement Network Function Virtualization (NFV) by offering each networking capability as a managed service. Instead of deploying physical routers, firewalls, or switches, we consume them on-demand using software-defined APIs.

üí° Key Insight: In the cloud, every network function is exposed as a service, and subnets act as logical boundaries where these services are applied.

üîπ AWS: Network Functions as Services

  • VPC ‚Äì Virtual network (software-defined data center network)
  • Subnets ‚Äì Logical segmentation inside VPC (public/private)
  • Internet Gateway ‚Äì Virtual device enabling internet access
  • NAT Gateway ‚Äì Managed outbound internet access for private subnets
  • Route Tables ‚Äì Software-defined traffic forwarding rules
  • Security Groups / NACLs ‚Äì Virtual firewalls
  • Load Balancer (ALB/NLB) ‚Äì Virtual traffic distribution
AWS View: Each subnet consumes NFV services through route tables, gateways, and security rules.

üîπ GCP: Network Functions as Services

  • VPC Network ‚Äì Global, software-defined virtual network
  • Subnets ‚Äì Region-specific logical partitions
  • Cloud Router ‚Äì Software-based routing control
  • Cloud NAT ‚Äì Managed NAT service
  • Firewall Rules ‚Äì Distributed virtual firewall
  • Cloud Load Balancing ‚Äì Software-based traffic management
GCP View: Networking is globally software-defined, with subnets acting as regional NFV zones.

üîπ Azure: Network Functions as Services

  • Virtual Network (VNet) ‚Äì Azure‚Äôs software-defined network
  • Subnets ‚Äì Isolation boundaries within VNet
  • Internet Gateway (Default) ‚Äì Built-in virtual connectivity

Azure View: Subnets are the attachment points where NFV services like NSG, NAT, and routing are enforced.

4Ô∏è‚É£ Load Balancing as a Network Function (AWS ALB & NLB)

‚åÑ

In traditional data centers, load balancers were expensive physical appliances. In the cloud, load balancing is delivered as a fully managed, software-defined network function‚Äîa classic example of NFV in action.

üí° Key Insight: In AWS, load balancers are not hardware devices. They are virtual network functions that distribute traffic intelligently using software logic.

üîπ Application Load Balancer (ALB)

Application Load Balancer operates at Layer 7 (Application Layer) of the OSI model. It understands application-level details such as URLs, HTTP headers, and request paths.

  • Content-based routing (URL / path-based routing)
  • Supports HTTP, HTTPS, WebSockets
  • Ideal for microservices and container-based applications
  • Integrates with ECS, EKS, and Kubernetes Ingress
Example: Requests to /login ‚Üí Auth Service, /orders ‚Üí Order Service
üìò Exam Line: Application Load Balancer works at Layer 7 and performs content-based routing.

üîπ Network Load Balancer (NLB)

Network Load Balancer operates at Layer 4 (Transport Layer) of the OSI model. It handles traffic based on IP address and port number with ultra-low latency.

  • Supports TCP, UDP, and TLS
  • Designed for high-throughput, low-latency workloads
  • Preserves source IP address
Example: Real-time gaming or financial transaction systems.
üìò Exam Line: Network Load Balancer works at Layer 4 and is used for high-performance workloads.
